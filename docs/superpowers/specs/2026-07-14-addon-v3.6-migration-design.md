# Addon Migration Design: Wealthfolio v3.5 -> v3.6.1+

## Context

This addon currently uses pre-sandbox patterns (runtime sidebar registration with React icon nodes, localStorage persistence, and Rollup globals for React). Wealthfolio 3.6+ runs addons inside an isolated iframe sandbox and requires updated routing, manifest metadata, build externalization, and storage APIs.

The target is a **declarative-complete migration** on top of 3.6.1+ behavior.

## Goals

1. Make the addon fully compatible with Wealthfolio 3.6.1+ sandbox behavior.
2. Move navigation source-of-truth to manifest contributions (`contributes.routes` and `contributes.links`).
3. Keep route rendering lifecycle host-managed via `component` API (3.6.1+).
4. Replace localStorage/sessionStorage usage with async `ctx.api.storage`.
5. Externalize host-provided dependencies as ESM and remove legacy React globals mapping.
6. Preserve current user-facing behavior and rebalance logic.

## Non-Goals

1. No algorithmic changes in rebalance calculations.
2. No unrelated UI redesign.
3. No backward compatibility mode for hosts older than 3.6.0.

## Chosen Approach

Adopt a declarative route/link setup in manifest while keeping runtime route renderer registration with strict id alignment.

- Manifest declares durable route and sidebar link identities.
- Runtime registers the renderer for the declared route id.
- Sidebar registration through runtime API is removed.
- Storage is migrated to async durable API.

This balances migration safety with long-term architecture clarity and aligns with Wealthfolio migration guidance.

## Architecture

### 1) Navigation and Routing

- Add route declaration in `contributes.routes` with stable id (e.g. `main`).
- Add sidebar link in `contributes.links.sidebar` that references route id `main`.
- Register runtime route using `ctx.router.add({ id: 'main', component: ... })`.
- Use exactly one route registration mode (`component`, not `render`) for 3.6.1+.
- Remove imperative `ctx.sidebar.addItem(...)` calls.

### 2) Runtime Composition

- Keep wrapper composition:
  - shared query client from host API
  - account selection provider
  - page component
  - toaster
- Keep lifecycle cleanup only for actual resources; no route-root unmount management needed for `component` mode.

### 3) Persistence Layer

- Replace `window.localStorage` reads/writes with `ctx.api.storage.get/set/delete`.
- Keep validation/fallback behavior to avoid crashes on malformed persisted data.
- Introduce async hydration and async write path in hooks using local state as immediate UI source.
- Ensure persistence failure does not block interactions.

## Component and File-Level Design

### `manifest.json`

- Bump `sdkVersion` to 3.6.x.
- Add `minWealthfolioVersion: "3.6.0"`.
- Add `hostDependencies` matching host-provided externals and SDK contract.
- Add `contributes.routes` and `contributes.links.sidebar`.
- Sidebar icon uses supported host icon name string.

### `src/addon.tsx`

- Remove `ctx.sidebar.addItem(...)` and related cleanup.
- Add route registration with stable `id` matching manifest route id.
- Use `component` property directly.
- Keep wrapper providers and page composition.

### `vite.config.ts`

- Remove `rollup-plugin-external-globals` usage.
- Externalize host-provided dependencies by bare ESM specifiers.
- Keep output format as ES module and existing entry/outDir.

### `package.json`

- Update `@wealthfolio/addon-sdk` and `@wealthfolio/ui` to 3.6.x ranges.
- Remove `rollup-plugin-external-globals` dependency.
- Keep scripts unless migration requires small updates.

### `src/lib/storage.ts` and related hooks

- Convert storage helpers to async API contract.
- Update hooks currently using localStorage-backed helper:
  - account selection persistence
  - tolerance persistence
  - plan persistence
- Keep behavior parity with current defaults and validators.

## Data Flow

1. Host discovers addon route/link from manifest.
2. User opens addon route from sidebar.
3. Runtime route component renders page via host-managed lifecycle.
4. Hooks hydrate persisted values from durable storage asynchronously.
5. UI updates once hydrated state is available; writes persist asynchronously on change.
6. Rebalance computation runs unchanged from hydrated/current state.

## Error Handling

1. Storage read/write failures:
   - log context
   - keep UI state in memory
   - avoid crashes
2. Invalid persisted payloads:
   - reject with validators
   - fallback to default values
3. Route id mismatch protection:
   - use shared constant/string source to avoid manifest/runtime divergence
4. Build/runtime compatibility:
   - enforce externalization of React and host libraries

## Testing Strategy

### Automated

1. Run unit tests and update async storage tests.
2. Run type-check for route config and manifest-aligned ids.
3. Build and verify no React bundling regressions.

### Manual migration checklist

1. Start addon dev mode and open the addon route.
2. Confirm page renders in sandbox (not blank).
3. Confirm sidebar entry is present via manifest contributions.
4. Update plan/tolerance/account, restart host, and verify persistence.
5. Disable/enable addon and confirm clean lifecycle with no route/root errors.
6. Validate no SecurityError from localStorage/sessionStorage usage.

## Risks and Mitigations

1. Async hydration timing can temporarily show defaults.
   - Mitigation: explicit hydrated flags and safe defaults.
2. Manifest/runtime route id mismatch causes blank page.
   - Mitigation: single shared id constant and test coverage.
3. Dependency range drift vs host guarantees.
   - Mitigation: align with SDK host dependency contract before release.

## Rollout Notes

1. This migration intentionally requires host 3.6.0+.
2. Release notes should mention storage backend migration and manifest-based navigation.
3. Verify packaged artifact includes updated manifest and ES build output.

## Acceptance Criteria

1. Addon loads and renders correctly on Wealthfolio 3.6.1+.
2. Sidebar navigation is declared through `contributes.links` and works.
3. Route is declared through `contributes.routes` and runtime renderer id matches.
4. No runtime use of localStorage/sessionStorage remains.
5. Build uses ESM externals for host dependencies; no React global mapping remains.
6. User plans/tolerance/account persistence still works after restart.
